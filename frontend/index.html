Dashboard en < 3 segundos
Con ~5,000 usuarios concurrentes, el dashboard de reportes debe responder en máximo 3s mediante caché Redis y agregación SQL.

✦ Agregación SQL — GROUP BY en DB, sin Python-side loops
✦ Índices de DB — company_id, project_id indexados
✦ Pool de conexiones — SQLAlchemy connection pool
⚡ ASR Escalabilidad
100% de éxito en extracciones
El extractor debe capturar métricas de forma agnóstica en ambientes sobrecargados, garantizando 100% de éxito mediante reintentos.

✦ Reintentos (MAX 5) — Backoff exponencial 2^n segundos
✦ Jitter aleatorio — Evita thundering herd en reintentos

✦ Celery async — Cola de tareas para extracciones pesadas

Log Masking — Datos Sensibles
ASR30 Seguridad — SensitiveDataFilter enmascara credenciales en logs de Django
🛡️ ASR30 Seguridad
100% de datos sensibles enmascarados en logs
El SensitiveDataFilter de Django intercepta todos los registros de log y reemplaza automáticamente AWS keys, JWTs, Account IDs y URIs de DB con [MASKED] antes de persistirlos. Cero datos sensibles en texto plano en los logs del sistema.

Disponibilidad — Failover MongoDB
Experimento de failover en clúster sharded · Criterio: delta_t < 5 segundos
🏗️ ASR Disponibilidad
Failover de shard MongoDB en menos de 5 segundos
Bajo carga sostenida de 200 req/s sobre GET /places/, el tiempo de failover al detener el nodo primario de un shard debe ser inferior a 5,000 ms en al menos 4 de 5 corridas. El throughput debe recuperarse al 100% en menos de 30 segundos post-fallo.

🧪 Experimento de Failover
1Inicia JMeter con jmeter_disponibilidad.jmx (200 req/s por 12 min)
2Espera 2 minutos de calentamiento
3En el minuto 5, detén el primario de shard1:
ssh ubuntu@IP "docker stop shard1a"
4Observa el pico de errores en JMeter y el tiempo de recuperación
5Reinicia el shard y repite 5 veces:
docker start shard1a && sleep 30
Criterio de aceptación
✓ delta_t failover < 5,000 ms en 4/5 corridas
✓ % error proporcional a delta_t × 200 req/s
✓ Throughput vuelve al 100% en < 30 segundos
